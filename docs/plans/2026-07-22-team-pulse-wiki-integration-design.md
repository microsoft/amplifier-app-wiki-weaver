# Team Pulse Wiki Integration Design

## Goal

Integrate the output of `wiki-weaver` into the Team Pulse product as a
native-rendered, deep-linkable `/wiki` surface that is:

1. **Automatically regenerated by CI** when the source corpus changes.
2. **Incrementally updated** (only new/changed sources are woven — not full
   rebuilds).
3. **A single grounded source of truth** serving BOTH AI agents (structured
   index/citation data) and humans (rendered pages) from the same artifacts.

Phase 1 ships a read-only `/wiki`. The design deliberately leaves clean seams
for a later **Phase 2 self-learning / admin-steering** layer, without building
any of it now.

## Background

Three sibling repositories under `team-pulse-structure/` participate in this
integration. Understanding what each owns is a prerequisite to the design.

### 1. wiki-weaver tool — `microsoft/amplifier-app-wiki-weaver`

- CLI entrypoint `wiki_weaver.cli:main`; commands include
  `init` / `ingest` / `ask` / `lint` / `build-dashboard`.
- `build-dashboard <wiki_dir> --out PATH` emits ONE self-contained HTML file
  (inline CSS/JS, client-side search).
- Ingest is strictly **SEQUENTIAL** — config `parallelism` is accepted but
  hard-pinned to `1`.
- Dedup / resumability via a content-hash ledger `.processed.jsonl` + registry
  `.sources.json`.
- Config `wiki.config.yaml` keys: `provider` (default `anthropic`), `models`
  per stage (`default` / `ingest` / `assess` / `feedback`), `max_cycles`
  (default `3`).
- Policy files under `.wiki/policy/`: `schema.md`, `convergence-rubric.md`,
  `validator.yaml`, `inner.dot`.
- Query indexes under `.wiki/index/*.json`: `backlinks.json`, `links.json`,
  `tags.json`, `properties.json`, `aliases.json`.
- Citations resolve via a page's frontmatter `sources:` ordinals →
  `.sources.json`.
- **Durable** (must persist between runs): `.wiki/`, the `*.md` pages,
  `.sources.json`, `.processed.jsonl`, `site.html`, `_archive/_sources/`.
- **Ephemeral** scratch (gitignore): `_inbox/`, `_failed/`, `.runs/`.

### 2. Data repo (`-made`) — `microsoft-amplifier/amplifier-app-team-pulse-made`

- Holds `data/corpus/{conversations (~48 md), repos (~3768 md)}`,
  `data/team-wiki/` (currently **UNTRACKED** in git), and `data/overlay/`.
- Already has `.github/workflows/` with a proven **commit-back** pattern in
  `daily-activity.yml` (git config bot identity → add → commit →
  rebase-and-retry push).
- `weekly-cycle.yml` shows the amplifier-tool-install CI pattern
  (`astral-sh/setup-uv` → `uv tool install git+...` → run, with
  `ANTHROPIC_API_KEY` secret).
- The app reads THIS repo **directly as a git "vault"** (env `VAULT_REPO_URL`,
  `VAULT_REPO_BRANCH=main`, `TEAM_PULSE_AUTO_PUSH=true`). Generated output MUST
  be committed to `-made` `main` to reach the app — **there is no blob push**.

### 3. App repo — `microsoft/amplifier-app-team-pulse`

- React 18 + Vite SPA (`frontend/`) served by FastAPI (`team_pulse/server.py`).
- Content-collection registry at `team_pulse/content/collection_registry.py`
  (`_CONTENT_COLLECTIONS`, entries like `corpus → data/corpus`), served by
  `/api/content/markdown?path=<rel>` (server.py ~1504-1525) and the lens
  `/api/lens/resources?collection=<name>` (routes_lens.py).
- Deep-link scheme `tp://doc/...` handled in `frontend/src/lib/entity-links.ts`.
- Existing route `/initiatives` → `InitiativesPage` registered at
  `frontend/src/App.tsx:149`, nav in `LeftNav.tsx:61-67`.
- The app ALSO already has a separate `/api/team-wiki` "Team Briefings" surface
  (`routes_team_wiki.py`, `TeamWiki.tsx`,
  `team_pulse/pipeline/wiki_weaver_engine.py`). **DECISION: leave that
  untouched; our new `/wiki` is separate.**
- Frontend already embeds self-contained HTML via `<iframe srcDoc>`
  (`KnowledgePage.tsx`) and renders markdown via ReactMarkdown (`TeamWiki.tsx`).

## Chosen Approach

**Native render — not iframe.** (Validated with the user.)

The rejected alternative was to embed the 2.7 MB self-contained `site.html` in
an `<iframe srcDoc>`. That does not scale:

- Monolithic — the blob only grows as the corpus grows toward the 3768 repo
  docs; the whole thing loads in one shot with no lazy-loading.
- **No per-page URLs** → deep-linking (the core requirement, the "connective
  tissue") is impossible. Cannot share/bookmark a page, cannot cross-link from
  other app surfaces, agents cannot address a page by URL.
- Walled off from the app's existing search, auth-gated collections, and
  styling.

Native render instead serves each page from structured data:

- Per-page routes (`/wiki/:slug`) → real deep-linking, shareable,
  agent-addressable, cross-linkable.
- **One source of truth** feeds both agents (`.wiki/index/*.json` +
  citation resolution) and humans (rendered pages).
- Lazy per-page loading → fine at 50 or 50,000 pages.
- Incremental: a corpus change updates a few pages, served immediately, with no
  monolith rebuild.
- Reuses the app's existing content-collection + `tp://` deep-link machinery.

**Cost accepted:** the app takes a dependency on the wiki's data schema
(frontmatter + `.wiki/index` format).

## Architecture

End-to-end flow:

```
corpus change (push to data/corpus/** in -made)
    │
    ▼
GH Actions: build-team-wiki.yml   (concurrency group → only one weave at a time)
    │
    ▼
wiki-weaver run   (resilient incremental batch driver)
    │  drains only NEW/CHANGED sources (content-hash dedup)
    ▼
build-dashboard   (regenerates site.html)
    │
    ▼
commit data/team-wiki/ back to -made main   (rebase-and-retry; no-op if no diff)
    │
    ▼
app reads -made as a git vault
    │
    ├─ collection_registry:  wiki → data/team-wiki
    ├─ /api/content/markdown?path=wiki/<slug>      → page markdown
    ├─ .wiki/index/*.json                          → backlinks / graph
    └─ /api/lens/resources?collection=wiki         → page list
    │
    ▼
frontend  /wiki/:slug   (native render)
```

Two consumer classes read the **same** artifacts:

- **Humans** — the `/wiki/:slug` frontend route renders page markdown.
- **Agents** — read `.wiki/index/*.json` + citation resolution directly. There
  is **no separate agent pipeline**.

## Components

### wiki-weaver: new `wiki-weaver run` subcommand

A **resilient incremental batch driver**. It packages what currently lives as
throwaway scripts so that CI and runtime share ONE implementation.

- Processes sources one at a time.
- Each source runs in a **fresh process** (fresh asyncio event loop) — this
  avoids the `RuntimeError: Event loop is closed` httpx teardown race that
  occurs when a network change severs an in-flight request.
- **Per-source timeout** → a non-converging source is quarantined to `_failed/`
  instead of wedging the whole run.
- **Resumable** via the existing content-hash dedup.
- After the batch, invokes `build-dashboard`.

### -made (data repo)

- `data/team-wiki/.gitignore` — the durable/ephemeral split.
- The tracked `data/team-wiki/` artifact (durable state committed).
- `.github/workflows/build-team-wiki.yml`.

### app backend

- Register a `wiki` content collection → `data/team-wiki` in
  `collection_registry.py`.
- Ensure `/api/content/markdown?path=wiki/<slug>` and
  `/api/lens/resources?collection=wiki` resolve it.
- A small endpoint (or reuse of an existing one) to serve `.wiki/index/*.json`
  for backlinks / graph.

### app frontend

- `/wiki/:slug` route + `WikiPage` component: fetch page markdown, render with
  ReactMarkdown, resolve `[[links]]` and `[^cite]`.
- Rename `/initiatives` → `/wiki` with a redirect from the old path.
- Nav update.

## Data Flow

In-page links resolve as follows:

- `[[wikilink]]` → `/wiki/:target`, resolved via `.wiki/index/links.json` +
  `aliases.json`.
- Citation `[^N]` → the exact source doc in `data/corpus/...` via
  `.sources.json`, rendered through the existing `tp://doc/...` /
  `/api/content/markdown` deep-link path.

Agents consume the SAME artifacts (`.wiki/index/*.json` + citation resolution)
— confirming the single-source-of-truth property end to end.

## Error Handling & Resilience

- **Per-source process isolation** → a fresh event loop per source roots out the
  `Event loop is closed` teardown race triggered by mid-request network changes.
- **Per-source timeout** → quarantine to `_failed/`; a single bad/slow source
  cannot stall the run, which continues.
- **Incremental content-hash dedup** → runs are resumable; a killed run
  re-continues from where it stopped.
- **CI concurrency** `concurrency: {group: team-wiki, cancel-in-progress:
  false}` → never two weaves racing the same wiki.
- **Commit-back** uses rebase-and-retry push (copied from `daily-activity.yml`)
  for race safety; a no-op when there is no diff.
- **GitHub Actions 6h job cap risk** → mitigated because CI only ever runs
  INCREMENTAL deltas (minutes); the large initial seed is built once
  out-of-band. This is WHY durable state must be committed.
- **Frontend** → a missing/renamed slug returns a graceful not-found; broken
  `[[links]]` (tracked in `.wiki/index` `_broken`) render as plain text, not
  dead links.

## Testing Strategy

**wiki-weaver `run`** — unit tests for the driver:

- Success path — source converges → ledger increments.
- Timeout path — slow source → quarantined to `_failed/`, run continues.
- Resume path — an already-ingested source is skipped via dedup.

**-made workflow:**

- `workflow_dispatch` smoke test.
- Assert commit-back is a no-op when nothing changed (exit 0) and commits when
  changed.
- Assert only `data/team-wiki/` durable files are staged (ephemeral dirs
  gitignored).

**app backend** — unit tests that:

- The `wiki` collection resolves.
- `/api/content/markdown?path=wiki/<slug>` returns a known page.
- `/api/lens/resources?collection=wiki` lists pages.

**app frontend:**

- `/wiki/:slug` renders a page.
- `[[wikilink]]` resolves to `/wiki/:target`.
- Citation `[^N]` resolves to the correct `data/corpus/...` doc via `tp://doc`.
- Redirect from `/initiatives` works.
- UPDATE the 6 existing tests that reference `/initiatives`: `App.test`,
  `LeftNav.test`, `InitiativesPage.test`, `ProjectDetailView.test`,
  `AdminRoute.test`, `use-post-login-return.test`.

**Integration** — end-to-end deep-link test: a wiki page citation resolves to
and opens the exact corpus source.

## Phased Plan

- **Phase 0 (DONE)** — initial seed wiki woven from
  `data/corpus/conversations` (45 sources converged, 6 quarantined);
  `site.html` rebuilt (73 pages).
- **Phase 0.5 (recommended, ~15 min)** — balance the seed: re-weave the 6
  quarantined sources (`base-camp-workstream-index`, `context-intelligence`,
  `overview`, `program-structure`, `resolvers`, `skunkworks-coordination`) and
  feed The Rig / Skunkworks / Program their child workstream docs so those pods
  accrete children and get promoted to `level: group`. Today only Base Camp is
  a real "group" because it had ~4 source docs vs one each for the others — a
  **data-balance issue, not a schema limit**.
- **Phase 1** — the two-repo native integration (below); ships a read-only
  `/wiki`.
- **Phase 2** — self-learning + admin steering (layers on top; see Reserved
  Seams).

## Reserved Seams for Phase 2

Do not build these now — just leave room:

- A committed `purpose` / `policy` config location the admin UI can later edit
  (the admin-configurable "default prompt" + policy knobs: `schema.md`,
  `convergence-rubric.md`, `validator.yaml`).
- The existing `data/overlay/` as the home for human corrections / pins that
  survive re-weaves.
- The runtime "force update" = an admin action triggers the same GH workflow via
  `workflow_dispatch` (no long-running job inside the app).
